# Accessibility (WCAG 2.2 AA)

This document describes the accessibility guardrail and the AA practices applied
to the **new screens + critical flows** of the Qeet ID admin and login frontends,
and how to extend the lint guardrail to the remaining screens incrementally.

## Scope

AA is enforced today on a deliberately scoped surface — not the full ~70-screen
catalog. The remaining screens are migrated incrementally (see
[Expanding coverage](#expanding-the-lint-coverage)). The currently-gated paths are:

**Admin — new screens**
- `src/routes/_app/auth/connections/oidc.tsx`
- `src/routes/_app/auth/connections/oidc.$clientId.tsx`
- `src/routes/_app/auth/connections/saml-idp.tsx`
- `src/routes/_app/auth/api/consent-grants.tsx`
- `src/routes/_app/auth/api/signing-keys.tsx`
- `src/routes/_app/access/check.tsx`
- `src/routes/_app/security/device-authorizations.tsx`
- `src/routes/_app/groups.$groupId.tsx`

**Admin — critical flows + app shell**
- `src/routes/_app/users/**`
- `src/routes/_app/auth/login-methods/**`
- `src/routes/_app.tsx` (app shell)
- `src/features/dashboard/components/{app-sidebar,nav-main,language-switcher}.tsx`

**Login app (Next.js)**
- `apps/qeetid-login/src/app/**/*.tsx`

## The lint approach

Linting is handled by [Biome](https://biomejs.dev/) configured at the workspace
root (`biome.json`). Biome's `recommended` preset enables the `a11y` rule group
globally (at warning severity for noisy rules, default severity for others).
This provides baseline accessibility coverage across all files without requiring
a scoped allowlist.

- The `a11y` rules enabled by Biome's recommended preset cover the most critical
  WCAG AA concerns (missing labels, autofocus, ARIA misuse, etc.).
- Individual rules can be downgraded per file using `// biome-ignore` comments
  with an explicit justification.
- Run `bun run check` from the repo root to verify all files.

## AA checklist applied

- **Skip link** — `_app.tsx` renders a visually-hidden-until-focused
  "Skip to main content" link as the first focusable element, targeting the
  `<main id="main-content" tabIndex={-1}>` content region (the `<Outlet/>` host).
- **Errors as alerts** — admin forms render errors through `@qeetrix/ui`
  `FieldError`, which already emits `role="alert"`. The login/consent/device
  forms use `<p role="alert">` directly. New admin screens follow suit.
- **Async / bulk feedback** — toasts (sonner) are announced automatically. For
  non-toast async output we added polite live regions:
  - `access/check.tsx` — the allow/deny result panel is wrapped in
    `aria-live="polite"`.
  - `components/data-table/bulk-bar.tsx` — the fan-out progress
    (`done / total processed…`) is a `role="status" aria-live="polite"` region.
- **Labels / inputs** — every `Input`/`Textarea` on the target screens has an
  associated visible `FieldLabel htmlFor` (or `<label htmlFor>` in the login
  app). Radix/Base-UI `Select` triggers have no native label association, so they
  are tied to their `FieldLabel` via `id` + `aria-labelledby` (or an
  `aria-label` on the trigger, as in `groups.$groupId.tsx`). Selection checkboxes
  carry an `aria-label`.
- **Tables** — header cells use `<TableHead>` (renders `<th>`) via `@qeetrix/ui`.
- **Icon-only controls** — header/shell icon buttons carry `aria-label`
  (e.g. command-palette, search, keyboard-shortcuts buttons in `_app.tsx`).
- **File upload** — `users/import.tsx` drop zone keeps the `<label htmlFor>`
  bound to the real `<input type="file" id>`; drag-and-drop handlers live on the
  wrapping element (progressive enhancement) rather than on the label.

## Dialogs, sheets, and focus management

`@qeetrix/ui` `Dialog`, `AlertDialog`, and `Sheet` are all built on
`@base-ui/react/dialog` (Base UI v1.5.0). Its `DialogPopup` uses
`FloatingFocusManager` with `modal` (focus trap), `initialFocus` (focus moves
into the dialog on open), `returnFocus`/`finalFocus` (focus returns to the
trigger on close), `restoreFocus: "popup"`, and built-in Escape-to-close. So
focus trap + return-focus + Esc are provided by the primitives — **no custom
focus management was added** (and none should be; do not rebuild them).

## Addressing lint findings

Run `bun run check` from the repo root to see all Biome diagnostics. For
intentional deviations (e.g. `autoFocus` on a single-task form), suppress
per-line with a justification comment:

```tsx
{/* biome-ignore lint/a11y/noAutofocus: single-task form guides users to the only field */}
<Input autoFocus ... />
```

Recommended approach for new screens: write the accessible markup first using
`@qeetrix/ui` primitives (they satisfy most rules), then run `bun run check`
and resolve any remaining Biome a11y warnings before opening a PR.
